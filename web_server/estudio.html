Principio de Control Exclusivo

En sistemas con control dual (físico y digital), la colisión de comandos ocurre cuando los Joysticks analógicos y la interfaz web envían órdenes a los motores simultáneamente. Esto causaría movimientos erráticos o sobrecargas eléctricas destructivas en el puente H.

Para prevenir esto, el firmware del Arduino Nano implementa un estado lógico exclusivo llamado manualMode (equivalente a controlWebActivo = !manualMode):
  • Modo Web (manualMode = false): Los joysticks físicos en A0, A1, A2 se ignoran por completo en el bucle principal. El Arduino solo acepta y ejecuta comandos que entren por el puerto serie USB (`CL`, `CR`, `EU`, `ED`, etc.).
  • Modo Manual (manualMode = true): El Arduino frena cualquier comando web de movimiento de motores. Solo lee los voltajes analógicos de los potenciómetros físicos para calcular velocidades relativas por PWM.
La transición entre modos requiere de una orden limpia (`MM` o `MW`) o de presionar el pulsador físico en el joystick de Elevación (debounced en pin D6). Al cambiar de modo, el Arduino obliga la detención inmediata de todos los motores (`stopAllMotors()`) para evitar derivas por comandos residuales en RAM.
Arduino Nano C++ Sketch (SoftwareSerial D12/D9 para logs inalámbricos)

El siguiente sketch configura un enlace serial alternativo por software en los pines **D12 (RX)** y **D9 (TX)**. Esto es muy útil en pruebas de ingeniería para transmitir trazas de telemetría o logs de estado de forma inalámbrica a través del ESP32 a 9600 baudios, mientras la conexión principal de control ocurre por USB (Serial de hardware).

#include <SoftwareSerial.h>

// Mapeo de Pines de Potencia y Señales
#define AIN1 2   // Motor Carro
#define AIN2 4
#define PWMA 3
#define BTN_MODE 6 // Botón de modo

// Configuración de SoftwareSerial solicitada en D12 (RX) y D9 (TX)
SoftwareSerial debugSerial(12, 9);

bool manualMode = false;
unsigned long lastCommandTime = 0;
const unsigned long TIMEOUT = 500; // Watchdog de seguridad (500 ms)

void setup() {
  pinMode(AIN1, OUTPUT);
  pinMode(AIN2, OUTPUT);
  pinMode(PWMA, OUTPUT);
  pinMode(BTN_MODE, INPUT_PULLUP);

  Serial.begin(9600);        // USB principal de alta prioridad
  debugSerial.begin(9600);   // Canal inalámbrico pasivo
  debugSerial.println("[SETUP] Canal de logs activado en Pines D12(RX)/D9(TX)");
}

void loop() {
  unsigned long now = millis();
  
  // Leer comandos desde USB
  if (Serial.available()) {
    String cmd = Serial.readStringUntil('\n');
    cmd.trim();
    processCommand(cmd);
    lastCommandTime = now;
  }

  // Watchdog por Software de Seguridad
  if (!manualMode && (now - lastCommandTime > TIMEOUT)) {
    stopAllMotors();
  }
}

void processCommand(String cmd) {
  if (cmd == "S") {
    stopAllMotors();
    debugSerial.println("[LOG] Motores detenidos por comando USB.");
  }
  // ... resto de comandos de control
}

void stopAllMotors() {
  digitalWrite(AIN1, LOW);
  digitalWrite(AIN2, LOW);
  analogWrite(PWMA, 0);
}
ESP32 MicroPython (Servidor Web asíncrono y Cola Circular FIFO de 50 líneas)

El ESP32 funciona como un **servidor de diagnóstico pasivo inalámbrico**. Su firmware levanta un punto de acceso (Access Point) y escucha datos por hardware serial desde el Nano. Para evitar problemas de saturación de memoria RAM (`OutOfMemoryException`), almacena los logs en una cola circular de tamaño máximo de 50 líneas.

import uasyncio as asyncio
from machine import UART, Pin

# Inicializar UART2 (RX=GPIO16, conecta al TX del SoftwareSerial de Arduino)
uart = UART(2, baudrate=9600, rx=16, tx=17)
led = Pin(2, Pin.OUT)

# Cola Circular (FIFO) de tamaño fijo para depuración segura en RAM
logs = []
MAX_LOGS = 50

def add_log(trace):
    global logs
    logs.append(trace)
    if len(logs) > MAX_LOGS:
        logs.pop(0) # Remueve el log más antiguo (FIFO)

async def uart_reader_task():
    print("Lector UART asíncrono activo...")
    buffer = b""
    while True:
        await asyncio.sleep_ms(30)
        if uart.any():
            chunk = uart.read()
            if chunk:
                buffer += chunk
                while b"\n" in buffer:
                    line, buffer = buffer.split(b"\n", 1)
                    try:
                        decoded = line.decode('utf-8').strip()
                        if decoded:
                            add_log(decoded)
                    exceptException:
                        pass

async def handle_client(reader, writer):
    try:
        request = await reader.read(1024)
        req_str = request.decode('utf-8')
        
        if req_str.startswith('GET / '):
            # Renderiza los logs de depuración (FIFO) en orden inverso
            log_lines = "".join([f'<div class="line">{l}</div>' for l in reversed(logs)])
            html = f"<html><body><h1>Logs de Grúa</h1>{log_lines}</body></html>"
            
            response = "HTTP/1.1 200 OK\r\nContent-Type: text/html\r\nConnection: close\r\n\r\n" + html
            await writer.awrite(response.encode('utf-8'))
    except Exception as e:
        pass
    finally:
        await writer.aclose()

async def main():
    asyncio.create_task(uart_reader_task())
    server = await asyncio.start_server(handle_client, '0.0.0.0', 80)
    led.value(1) # Encender LED indicando que el AP está listo
    await server.wait_closed()

asyncio.run(main())
Navegador Web JavaScript (Uso Asíncrono de Web Serial API sin bloqueo)

La **Web Serial API** permite al sitio web conectarse directamente al Arduino Nano usando el puerto USB de la laptop. Al ejecutarse asíncronamente con Promises y Async/Await, la comunicación de tramas JSON ocurre de fondo, evitando congelar la interfaz de usuario.

// Solicitar puerto y establecer conexión asíncrona
async function connectSerial() {
  try {
    // 1. Abre el cuadro de diálogo nativo del navegador
    const port = await navigator.serial.requestPort();
    await port.open({ baudRate: 9600 });
    
    console.log("Conexión USB establecida con éxito.");
    
    // 2. Iniciar bucle de lectura sin bloquear el hilo principal (Event Loop)
    readSerialLoop(port);
  } catch (err) {
    console.error("Fallo al abrir el puerto serie:", err);
  }
}

// Envía comandos simples de forma asíncrona
async function writeCommand(port, commandText) {
  if (!port || !port.writable) return;
  const encoder = new TextEncoder();
  const writer = port.writable.getWriter();
  
  // Transmitir comando terminado en salto de línea para Arduino
  await writer.write(encoder.encode(commandText + "\n"));
  writer.releaseLock();
}

Divisor de Tensión para Comunicación de 5V a 3.3V

El Arduino Nano opera con lógica digital de 5V, lo que significa que su pin de salida D13 (TX de SoftwareSerial) emitirá señales con amplitud de 5V. Sin embargo, el microcontrolador ESP32 funciona con una lógica de 3.3V y sus pines de entrada (como el RX2, GPIO 16) no toleran voltajes superiores a 3.6V. Conectarlos directamente destruiría permanentemente el pin del ESP32 o el chip completo.

Para solucionar esto, se implementa un **Divisor de Tensión Resistivo** compuesto por dos resistencias ($R_1 = 1k\Omega$ y $R_2 = 2k\Omega$) conectadas en serie, que atenúan la amplitud de la señal de 5V a un nivel seguro de 3.33V.

[ Arduino Nano TX (5V) ] | | [R1] (1k Ohmios) | +--------------> [ ESP32 RX2 (GPIO 16) ] (Señal segura a 3.33V) | [R2] (2k Ohmios) | | [ GND ] (Tierra Común del Sistema)
Fórmula Matemática del Divisor:
$$V_{out} = V_{in} \cdot \frac{R_2}{R_1 + R_2}$$
Sustituyendo los valores de diseño empleados en el prototipo:
$$V_{out} = 5V \cdot \frac{2000\Omega}{1000\Omega + 2000\Omega} = 5V \cdot \frac{2}{3} = 3.33\text{ V}$$ El voltaje resultante ($3.33\text{V}$) es ideal porque supera el umbral de detección de nivel lógico alto del ESP32 ($V_{IH} \approx 2.4\text{V}$) sin sobrepasar el límite de protección del pin ($3.6\text{V}$).

Aislamiento Crítico de Alimentación (Brownout Prevention)

Uno de los problemas más frecuentes en la robótica educativa es el reinicio constante ("brownout") del microcontrolador al arrancar los motores. Cuando un motor de corriente continua arranca, exige momentáneamente una corriente de pico que puede multiplicar por 3 su corriente de consumo nominal (Corriente de Arranque o "Stall Current").

Si los circuitos lógicos del Arduino y la potencia de los motores comparten la misma fuente (por ejemplo, el puerto USB de la laptop o una batería básica de 9V), esta caída súbita de voltaje hará que la línea de alimentación digital baje de los 4.5V mínimos requeridos por el Nano, provocando que el oscilador se apague y la grúa se reinicie en bucle infinito.

Para mitigar esto, el diseño eléctrico de la grúa separa completamente los dos ramales de corriente:

  • Línea Lógica (5V - USB): El puerto USB de la laptop alimenta exclusivamente al Arduino Nano, sensores, LEDs y la lógica interna del puente H TB6612FNG (pin VCC). Esto garantiza un voltaje libre de ruido térmico o interferencia inductiva.
  • Línea de Potencia (9V a 12V - Fuente Externa 2A): Conectada directamente al terminal VM (Motor Voltage) del puente H TB6612FNG y de allí a los bobinados de los motores. Es capaz de suministrar hasta 2 amperios en picos sin afectar la alimentación del microcontrolador.

Importante: Para que las señales lógicas de control del Arduino lleguen correctamente a los transistores del puente H, se deben conectar físicamente **las tierras comunes (GND)** de ambas fuentes en un único nodo.

Arquitectura de Comunicación Dual

El prototipo destaca por su topología de comunicación híbrida, combinando un canal de cableado determinista y un enlace de depuración inalámbrico:

  1. Canal Físico USB (Control de Alta Prioridad): Enlace directo por cable USB a 9600 bps usando la **Web Serial API**. Garantiza una latencia mínima (< 10ms) y es inmune a las interferencias electromagnéticas del aire. Acepta los comandos de control del operador y transmite telemetría instantánea (como el ángulo de giro calculado en tiempo real).
  2. Canal Wi-Fi (Diagnóstico e Inspección de Baja Prioridad): El Arduino Nano replica todos sus mensajes de depuración a través de `SoftwareSerial` hacia el ESP32. El ESP32 funciona como un servidor web HTTP que aloja una terminal retro. El usuario puede conectarse con su celular al SSID AP `ESP32-Grua-Setup` para revisar qué comandos se han ejecutado, fallos detectados o registros de depuración sin perturbar el canal USB de control principal.

Cuadrícula Comparativa de Enlaces de Red

Web Serial API (USB)
  • Latencia: Extremadamente baja (< 5 ms).
  • Estabilidad: Máxima. Inmune a interferencias de radiofrecuencia (RF).
  • Instalación: Cero drivers. Ejecución directa en Google Chrome / MS Edge.
  • Seguridad: Física. Requiere acceso local al puerto USB.
Web Sockets (Wi-Fi)
  • Latencia: Baja a media (15-50 ms según calidad del enlace inalámbrico).
  • Estabilidad: Variable. Depende de la atenuación del aire e interferencias en banda 2.4GHz.
  • Conexión: Inalámbrica bidireccional continua (Full-Duplex).
  • Consumo: Elevado debido a la potencia del transceptor Wi-Fi.
HTTP / REST (Wi-Fi)
  • Latencia: Media a alta (50-200 ms por cabeceras y Handshake TCP).
  • Estabilidad: Adecuada para lecturas esporádicas y telemetría pasiva.
  • Protocolo: Unidireccional impulsado por cliente (Request-Response).
  • Uso en Grúa: Utilizado en el celular para leer la cola circular de logs UART.

Flashcards Giratorias 3D de Conceptos Clave

1. Web Serial API

¿Cómo interactúa la web con el Arduino sin usar un servidor local?

Es una API de JavaScript nativa de navegadores Chromium que permite al navegador abrir canales directos de lectura y escritura con dispositivos serie USB (como Arduino) de forma asíncrona, eliminando la necesidad de instalar Node.js u otros servidores locales en la laptop de control.

2. SoftwareSerial

¿Qué limitaciones tiene y para qué sirve en este prototipo?

Es una biblioteca que emula un puerto serie por software en pines digitales normales. Se utiliza para redirigir los logs de depuración hacia el ESP32, dejando libre el puerto UART nativo (pines D0/D1) del Arduino, evitando cortocircuitos al cargar firmware por cable USB.

3. Estabilidad ESP32

¿Cómo mitigamos errores OutOfMemory en el servidor MicroPython?

MicroPython maneja memoria RAM limitada. Mitigamos fugas usando un buffer circular FIFO rígido limitado a 50 líneas de registros y liberando los sockets de red HTTP de forma inmediata al responder (`writer.aclose()`), forzando al recolector de basura (`gc.collect()`) en intervalos.

Mecanismo de Seguridad: Watchdog por Software

En el control inalámbrico o cableado de maquinaria pesada, un fallo de conexión o la desconexión física del cable USB del operador con los motores encendidos podría causar un desastre (por ejemplo, el motor del carro moviéndose de forma indefinida hacia el límite físico de la pluma).

Para prevenir esto, se implementa un **Watchdog por Software** (Perro guardián de inactividad):
  1. Latido Periódico (Heartbeat): La interfaz web del navegador envía automáticamente un carácter o comando leve cada 250 milisegundos al Arduino Nano.
  2. Temporizador en Arduino: El Arduino mantiene actualizado un contador de tiempo (`lastCommandTime = millis()`) cada vez que recibe un comando válido por serie.
  3. Apagado de Emergencia (Timeout): En cada ciclo del bucle `loop()`, el Arduino evalúa la diferencia de tiempo: `if (millis() - lastCommandTime > 500)`. Si transcurren **más de 500 ms** sin recibir comunicación del navegador, el Arduino asume pérdida de enlace y de inmediato corta el paso de energía a todos los motores (`stopAllMotors()`).

Retos Prácticos de Ingeniería (Defensa Técnica)

En el tribunal de defensa del proyecto de grado es común que propongan modificaciones físicas o de código sobre los planos de schema.html. Repasa a continuación 3 retos prácticos clave:

Reto 1: Reasignar el pulsador físico de modo del Joystick de A3 a D13
Implicaciones Electrónicas y Solución:

1. **Implicación Electrónica**: El pin digital D13 del Arduino Nano está conectado internamente en la placa a un diodo LED SMD de estado con una resistencia limitadora. Al usar D13 como entrada digital con resistencia pull-up interna (`INPUT_PULLUP`), la presencia del LED integrado en la línea actúa como una carga en paralelo a tierra, lo que puede provocar que el pin nunca registre un estado lógico ALTO completo (se quede flotando en alrededor de 1.7V a 2.5V en lugar de los 5V ideales). Para solucionarlo de forma robusta, se prefiere soldar un **resistor pull-up físico externo de 4.7kΩ** conectado a VCC.

2. **Modificación del código de Arduino C++**:

// Antes: #define BTN_MODE A3 o D6
#define BTN_MODE 13 // Nueva asignación física

void setup() {
  // Configuración con resistencia pull-up de seguridad en pin D13
  pinMode(BTN_MODE, INPUT_PULLUP);
}
Reto 2: Reasignar el TX de SoftwareSerial al pin D10 y modificar el divisor de tensión
Implicaciones Electrónicas y Solución:

1. **Implicación Electrónica**: Al reasignar el canal de depuración por software, la señal de transmisión de 5V ahora sale por D10. Debemos desconectar físicamente la línea del pin antiguo D13 y soldar la entrada del divisor de tensión en D10. Los valores de las resistencias del divisor de tensión **no necesitan modificarse** (1kΩ y 2kΩ se mantienen), ya que la amplitud de salida lógica en D10 sigue siendo de 5V en un Arduino Nano. Sin embargo, hay que tener extremo cuidado con no colisionar con la línea de transmisión USB del programador SPI o puerto de carga de firmware.

2. **Modificación del código de Arduino C++**:

#include <SoftwareSerial.h>
// Antes: SoftwareSerial debugSerial(10, 13); (donde 13 era TX)
// Ahora: Asignar D9 como RX (no usado) y D10 como TX
SoftwareSerial debugSerial(9, 10); 

void setup() {
  debugSerial.begin(9600);
}
Reto 3: Mover el pin PWM de Elevación de D6 a D3 e intercambiarlo con el Carro
Implicaciones Electrónicas y Solución:

1. **Implicación Electrónica**: El ATmega328P del Arduino Nano agrupa sus salidas PWM en temporizadores de hardware internos (Timer0: D5, D6; Timer1: D9, D10; Timer2: D3, D11). El Timer0 también controla funciones nativas del microcontrolador como `millis()` y `delay()`. Mover el motor de Elevación a D3 e intercambiarlo con el Carro a D6 libera al Carro de posibles interferencias del Timer0 y redistribuye las frecuencias PWM. En el cableado del prototipo, debemos conmutar físicamente los cables de señal lógica hacia los pines del puente H TB6612FNG (PWMA y PWMB).

2. **Modificación del código de Arduino C++**:

// Antes: PWMA (Carro) = D3, PWMB (Elevación) = D5 o D6
// Intercambio físico y lógico de pines PWM de control de velocidad:
#define PWMA 6 // Motor Carro ahora en D6 (Timer0)
#define PWMB 3 // Motor Elevación ahora en D3 (Timer2)

void setup() {
  pinMode(PWMA, OUTPUT);
  pinMode(PWMB, OUTPUT);
}

Banco Completo de Preguntas Teóricas y de Diseño

Banco interactivo colapsable con buscador en tiempo real y filtro por categoría temática. Excelente para que los integrantes del grupo ensayen respuestas antes de la ronda de preguntas individuales.

T1. ¿Qué significa configurar la velocidad serie a 9600 baudios en el sistema?

Significa que se transmiten aproximadamente 9600 bits por segundo a través de la línea de datos. Esta velocidad es ideal para entornos ruidosos en robótica porque asegura un bajo índice de error de trama y es perfectamente compatible tanto con la emulación por software en el Nano como con el UART por hardware del ESP32.

T2. ¿Qué navegadores soportan nativamente la Web Serial API?

Es soportada por navegadores basados en el motor Chromium, tales como Google Chrome, Microsoft Edge y Opera. Navegadores como Firefox y Safari no la implementan activamente por políticas internas de seguridad respecto al acceso directo a hardware del cliente.

T3. ¿Qué ventajas tiene el puente H TB6612FNG frente al clásico L298N?

El TB6612FNG está basado en transistores MOSFET, lo que le confiere una caída de voltaje interna bajísima y una alta eficiencia térmica. El L298N usa transistores bipolares que disipan mucha energía en forma de calor y restan cerca de 1.5V a 2V a los motores.

T4. ¿Por qué es vital usar uasyncio en el firmware del ESP32?

Porque el ESP32 debe realizar dos tareas concurrentes de forma continua: atender solicitudes HTTP web en el puerto 80 y leer tramas por la UART. Usar sockets tradicionales de red bloquearía la CPU del ESP32 esperando datos, deteniendo la lectura del puerto serie. Con `uasyncio`, las tareas ceden el control mientras esperan datos, permitiendo multitarea cooperativa fluida.

T5. ¿Qué sucede si olvidamos unificar las tierras (GND) lógica y de potencia?

Si no se conectan los terminales GND de ambas fuentes a un punto común, las señales de control PWM enviadas por el Arduino Nano hacia los pines de entrada del puente H no tendrán una referencia estable de retorno. Esto causará que los motores no enciendan o se comporten de forma errática debido al ruido digital.

T6. ¿Qué es el "debounce" y por qué se requiere en el pin del pulsador D6?

Los pulsadores mecánicos tienen laminas metálicas internas que rebotan microscópicamente durante 10 a 50 ms al ser presionados. Para el microcontrolador (que corre a 16 MHz), esto parece como si el usuario presionara el botón cientos de veces por segundo. El debounce por software (usando `millis()`) filtra estos rebotes falsos exigiendo que el estado se mantenga estable por 50 ms antes de validar la pulsación.

T7. ¿Cómo funciona el control PWM en los motores de corriente continua?

La modulación por ancho de pulso (PWM) conmuta el encendido y apagado de la señal de alimentación a una frecuencia muy alta (normalmente a 490Hz o 980Hz). Modificando la relación entre el tiempo encendido (ON) y apagado (OFF) en el rango de 0 a 255 ("Duty Cycle" o ciclo de trabajo), se regula el valor de tensión promedio efectivo que recibe el devanado del motor, controlando su velocidad con precisión.

T8. ¿Qué es la "Zona Muerta" analógica y para qué sirve en el Joystick?

Los Joysticks potenciómetricos físicos rara vez regresan a su punto medio exacto de tensión (2.5V o valor ADC 512) debido a tolerancias del resorte interno y vibración ("drift"). La zona muerta es un umbral lógico configurado en el código (por ejemplo, ±80 unidades de lectura digital) en el cual no se genera ningún comando de movimiento. Esto previene vibraciones y movimientos lentos indeseados de los motores en reposo.

T9. ¿Por qué transmitimos telemetría en formato JSON por puerto serie?

Porque JSON es un formato estándar, auto-descriptivo y extremadamente fácil de analizar ("parsear") en JavaScript nativo del navegador mediante `JSON.parse()`. Esto permite enviar estructuras complejas como `{"angle": 45, "mode": "web"}` que el navegador puede mapear a variables visuales directamente en una sola línea de código.

T10. ¿Qué limitaciones tiene la biblioteca SoftwareSerial frente a la UART nativa?

SoftwareSerial consume tiempo de procesador en el bucle principal porque la CPU del microcontrolador debe leer/escribir cada bit mediante interrupciones de temporización. Además, no es recomendable utilizar tasas de baudios superiores a 38400 bps debido a la pérdida de sincronía por interrupciones concurrentes de otros temporizadores.

T11. ¿Por qué se prefiere el puerto serie físico sobre conexiones inalámbricas para control de la grúa?

Por determinismo operacional y latencia. Un enlace físico USB tiene tiempos de respuesta inferiores a los 5 milisegundos y un enlace garantizado 100% libre de congestión espectral, factores indispensables al manipular carga suspendida en una grúa real.

T12. ¿Cuál es la diferencia entre modo Access Point (AP) y modo Station (STA) en el ESP32?

En modo Access Point (AP), el ESP32 emite su propia señal de red inalámbrica y asigna direcciones IP a los celulares que se le conecten (como la IP por defecto 192.168.4.1). En modo Station (STA), el ESP32 actúa como un cliente que busca y se conecta a un router Wi-Fi doméstico o red externa preexistente.

T13. ¿Por qué se usa pull-up interna en lugar de dejar el pin del pulsador flotando?

Si un pin digital configurado como entrada se queda flotando sin ninguna resistencia que lo ancle a VCC o a GND, actuará como una antena microscópica sensible al ruido electromagnético ambiental de la red eléctrica u otros cables del motor. Esto provocaría lecturas falsas de activación. El pull-up lo ancla a 5V de forma predeterminada asegurando un estado lógico estable hasta que el pulsador cierre el circuito a masa.

T14. ¿Cuál es la función del pin STBY (Standby) del puente H TB6612FNG?

El pin STBY apaga por completo las etapas de potencia del puente H, reduciendo el consumo de corriente a niveles ínfimos y liberando los transistores para dejar los motores sueltos. Debe estar conectado a un pin digital en estado lógico ALTO (5V) para activar el controlador de motores.

T15. ¿Cómo calcula el Arduino el ángulo de rotación de la pluma sin encoder físico?

Mediante integración numérica en base a tiempo: sabemos que a velocidad de PWM máxima (255) el motor del engranaje gira a 30 RPM (o 180° por segundo). En el `loop()`, se calcula el diferencial de tiempo (`dt` en milisegundos) transcurrido entre iteraciones y se multiplica por el factor de velocidad de PWM actual, acumulando el resultado en una variable flotante de ángulo de 0 a 360 grados.

T16. ¿Qué significa que una cola de datos sea de tipo FIFO?

FIFO significa "First-In, First-Out" (Primero en Entrar, Primero en Salir). En nuestra cola de depuración en MicroPython, al alcanzar el límite de 50 registros en RAM, la inserción de una nueva traza de logs obliga a la cola a descartar automáticamente la traza más vieja en base al índice cero (`logs.pop(0)`), manteniendo la consola de diagnóstico actualizada sin devorar la memoria del ESP32.

T17. ¿Qué propiedades CSS clave habilitan el efecto estético de glassmorphism?

Las propiedades fundamentales son: un fondo de color semitransparente (ej. `background: rgba(255, 255, 255, 0.1)`), un borde muy sutil y claro para acentuar el borde de cristal, y sobre todo la propiedad **`backdrop-filter: blur(12px)`** que se encarga de difuminar de forma nativa todo el contenido gráfico situado por detrás del elemento.

T18. ¿Qué es un condensador de desacoplo y cómo ayuda a estabilizar la potencia?

Es un condensador electrolítico de alta capacidad (ej: 470µF a 1000µF) conectado en paralelo a los terminales de alimentación del puente H. Actúa como un pequeño acumulador de energía instantánea de alta velocidad: cuando el motor arranca y exige corriente súbita, el condensador descarga su energía mitigando la caída local de voltaje en la línea y protegiendo al microcontrolador de oscilaciones destructivas en la tensión.

T19. ¿Qué ocurre si la tasa de baudios del Arduino y del ESP32 no coincide?

Ocurre un desajuste de sincronización temporal ("baudrate mismatch"). El receptor leerá los flancos de subida y bajada de voltaje en momentos erróneos, lo que deformará los caracteres de la palabra y provocará que el ESP32 reciba cadenas indescifrables de caracteres extraños o basura digital.

T20. ¿Qué diferencia hay entre un Watchdog de hardware y el de software programado?

El Watchdog de hardware es un periférico interno del microcontrolador conectado al circuito de Reset. Si la CPU se congela físicamente por código atascado, el temporizador de hardware expira y reinicia la placa completa eléctricamente. El Watchdog por software que implementamos es una condicional lógica del programa dedicada exclusivamente a validar de forma segura el estado de un enlace remoto para apagar actuadores, sin forzar un reinicio del microcontrolador.

R1. Reto: Calibración dinámica del centro del Joystick en Setup

Problema de Diseño: La grúa en ocasiones se activa sola en reposo porque el centro del joystick no lee exactamente 512, sino 530 o 490 por desgaste térmico.

Solución por Firmware: En la función `setup()`, con los joysticks en absoluto reposo, se promedian 10 lecturas ADC consecutivas para capturar el error de compensación nativo ("offset"). Este valor capturado reemplaza al valor estático de "512" en los mapeos analógicos de la grúa del bucle `loop()`, asegurando un balance en reposo dinámico perfecto.

R2. Reto: Inversión física y lógica de la dirección de un motor

Problema de Diseño: Al presionar el botón "Girar Izquierda", la pluma gira físicamente a la derecha por un error en la polaridad del ensamblado de cables.

Solución A (Física): Desconectar físicamente el par de cables de salida del motor C del puente H e invertirlos en sus respectivas clemas.

Solución B (Lógica): Modificar en el código Arduino las líneas dentro de `setMotorC()` permutando los estados altos y bajos del puente H: donde escribe `digitalWrite(CIN1, HIGH)` cambiarlo a `LOW` y viceversa.

R3. Reto: Límites de carrera virtuales mediante estimación de ángulo

Problema de Diseño: La grúa no debe girar libremente más allá de los 270° porque el cableado interno se enrolla y podría romperse.

Solución de Software: Con base al ángulo calculado en tiempo real, programar una condicional rígida en el firmware de control: `if (estimatedAngle >= 270.0 && speed > 0) { speed = 0; }`. De esta manera, si la pluma intenta sobrepasar el ángulo límite en dirección a las agujas del reloj, el paso al motor de giro se corta inmediatamente a nivel de software.

R4. Reto: Diagnóstico físico de enlace serie cortado

Problema de Diseño: La red inalámbrica del ESP32 está activa y la consola web carga en el smartphone, pero se queda eternamente en: "Esperando logs de depuración del Arduino Nano..." sin mostrar nada.

Procedimiento de Diagnóstico:
1. **Inspección Física**: Con el multímetro, verificar continuidad eléctrica en el cable de señal entre el Pin D13 del Nano y la entrada del divisor de tensión.
2. **Medición de Voltaje**: Comprobar con el osciloscopio o multímetro que la caída en el punto intermedio del divisor resistivo de tensión marque 3.3V al enviar tramas y no 0V permanentes o 5V dañinos.
3. **Verificación de GND**: Asegurar que la masa del Arduino Nano esté unificada con el terminal GND del ESP32.
